# Heavy Design System — User Stories

## Token Pipeline

### US-1: Generate CSS custom properties from token JSON
- [x] **As a** design system maintainer,
**I want to** generate CSS custom properties from shared token JSON files,
**So that** all projects consuming the design system get consistent, up-to-date token values.

**Given** token JSON files exist in `~/Projects/design/tokens/` (base and alias directories)
**When** I run `npm run build:tokens`
**Then** `dist/tokens.css` is generated with all base and alias tokens as CSS custom properties, including colors, scale, spacing, radius, and typography tokens.

---

### US-2: Support project-specific token overrides
- [x] **As a** design system maintainer,
**I want to** pass a project overrides file to the token build,
**So that** consuming projects can customize specific alias tokens without forking the system.

**Given** a project has an `alias-overrides.json` file
**When** I run `npm run build:tokens -- --project-overrides=/path/to/overrides.json`
**Then** the override values are deep-merged on top of the standard alias tokens, and the resulting CSS reflects the overridden values.

---

### US-3: Resolve token references in the build pipeline
- [x] **As a** design system maintainer,
**I want to** token references (e.g., `{gray.50}`) to resolve to actual values in the style guide pages,
**So that** the documentation shows resolved hex colors and pixel values, not raw reference syntax.

**Given** alias tokens reference base tokens using `{family.stop}` syntax
**When** I run `npm run build:pages`
**Then** the build resolves references through the lookup map, displaying actual hex values and pixel sizes in the style guide tables.

---

## Theming

### US-4: Support light and dark themes via data attribute
- [x] **As a** developer,
**I want to** toggle between light and dark themes using a `data-theme` attribute,
**So that** I can implement theme switching without reloading CSS.

**Given** `main.css` defines both `:root` (light) and `[data-theme="dark"]` token overrides
**When** I set `data-theme="dark"` on the `<html>` element
**Then** all UI color tokens (background, surface, border, text, action) switch to their dark theme values.

---

### US-5: Generate heavy-theme.css for Figma plugins
- [x] **As a** design system maintainer,
**I want to** generate a combined theme CSS file for Figma plugins,
**So that** all plugins share a consistent look without manually maintaining theme files.

**Given** theme variable and component files exist in `src/themes/heavy-plugin/`
**When** I run `npm run build:themes`
**Then** `dist/heavy-theme.css` is generated by concatenating variables and components, and automatically copied to `keith-templates/figma-plugin/heavy-theme.css` for all plugins to consume via symlink.

---

## Style Guide Pages

### US-6: Generate foundation pages from token data
- [x] **As a** design system maintainer,
**I want to** foundation pages (Colors, Typography, Spacing, Radius, Scale, etc.) to be auto-generated from token JSON,
**So that** documentation stays in sync with actual token values without manual updates.

**Given** token JSON files contain color families, type scale, spacing, and radius values
**When** I run `npm run build:pages`
**Then** HTML pages are generated in `dist/` with auto-populated tables showing token names, CSS variables, and resolved values — including color swatches for each family and stop.

---

### US-7: Display color tokens with light and dark theme comparison
- [x] **As a** developer,
**I want to** see UI color tokens side by side for light and dark themes,
**So that** I can verify that both themes have appropriate contrast and token mappings.

**Given** light and dark theme token files exist
**When** the Colors page is generated
**Then** UI color tables show each token with both light and dark hex values, color swatches, and their reference paths to base tokens.

---

### US-8: Generate component documentation pages
- [x] **As a** design system maintainer,
**I want to** generate component pages with live previews and code snippets,
**So that** developers can see how to use each component correctly.

**Given** component content functions are defined in `build-pages.js`
**When** I run `npm run build:pages`
**Then** each component gets an HTML page following the canonical section arc (Intro, Anatomy, Dimensions, Props, Guidelines, Accessibility, Related) with playground previews and copyable code blocks.

---

### US-9: Navigate between pages via sidebar
- [x] **As a** developer,
**I want to** browse all foundations, components, and patterns through a sidebar navigation,
**So that** I can quickly find the documentation I need.

**Given** the style guide is served
**When** I view any page
**Then** a sticky sidebar shows all pages grouped by category (Foundations, Components, Patterns) with the active page highlighted, and clicking a link navigates to that page.

---

## CSS Architecture

### US-21: Consume design system via single CSS import
- [x] **As a** developer,
**I want to** import a single `main.css` file to get the full design system,
**So that** I don't have to manage multiple CSS imports or build steps.

**Given** `dist/main.css` contains all tokens, reset, typography, grid, layout, and components
**When** I import or link `main.css` in my project
**Then** all design system tokens and component styles are available via CSS custom properties and class names.

---

### US-22: Use HDS-prefixed component classes
- [x] **As a** developer,
**I want to** all component classes to use the `hds-` prefix,
**So that** design system styles never conflict with project-specific CSS.

**Given** component styles are defined in main.css
**When** I use design system components
**Then** all classes use the `hds-` prefix (e.g., `hds-btn`, `hds-form-input`, `hds-card`, `hds-select`, `hds-badge`).

---

## Build System

### US-23: Full build pipeline in one command
- [x] **As a** design system maintainer,
**I want to** run a single build command that generates everything,
**So that** I can rebuild the entire system after any change.

**Given** source files and token JSON are in place
**When** I run `npm run build`
**Then** it runs all four build steps in sequence: tokens, styles, themes, and pages — producing a complete `dist/` output.

---

### US-24: Watch mode with live reload for development
- [x] **As a** design system maintainer,
**I want to** a development mode that watches for changes and live-reloads the browser,
**So that** I can see updates instantly while working on the style guide.

**Given** I'm developing the design system
**When** I run `npm run dev`
**Then** a file watcher monitors source files for changes, triggers rebuilds automatically, and Browser-Sync serves the style guide with live reload on port 3100.

---

### US-25: Copy style-guide framework CSS into dist
- [x] **As a** design system maintainer,
**I want to** the build to concatenate the shared style-guide framework CSS with the design system's extension CSS,
**So that** the style guide pages have both the framework chrome and design-system-specific documentation styles.

**Given** `style-guide/src/style-guide.css` and `src/styles/heavy-style-guide-ext.css` exist
**When** I run `npm run build:styles`
**Then** `dist/style-guide.css` is generated by concatenating the framework CSS and the extension CSS, and `main.css` is copied to `dist/`.
